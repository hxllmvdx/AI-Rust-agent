# REFLECTION

## Почему я выбрал именно эту задачу

Я выбрал комбинацию **Track A + C**, потому что она позволяла показать два разных типа инструментов. Первый инструмент дает **живые данные** через GitHub API, то есть агент может опираться не только на заранее известные сведения, но и на актуальную информацию о репозиториях. Второй инструмент работает с **локальной структурированной базой знаний** в JSON-формате. Это было важно, потому что одних live-данных недостаточно для осмысленного сравнения технологий: GitHub API хорошо показывает репозитории, активность, язык, stars и обновления, но не отвечает на вопрос о компромиссах, типичных use cases, архитектурных trade-offs и практических плюсах и минусах.

Мне также было важно сделать проект на **Rust** как основной backend-стек. Это дало возможность явно разделить архитектуру на planner, orchestrator, tools, synthesizer, session storage и API-слой. Отдельно мне понравилась идея хранить сессии в Redis, а не в памяти процесса: даже для маленького проекта это сразу делает систему более “настоящей”, потому что появляется внешнее хранилище состояния и исчезает привязка к одному процессу. Для frontend я выбрал Streamlit, потому что хотел быстро получить рабочий интерфейс и сосредоточиться не на верстке, а на агентной логике и интеграции компонентов.

В итоге задача оказалась для меня интересной именно как инженерная: здесь нужно было думать не только о prompt’ах, но и о том, как грамотно соединить LLM, API, локальные данные, сессии, Docker Compose и пользовательский интерфейс в одну работающую систему.

## Где агент ломался и что получилось неидеально

Самый заметный failure case был связан с **planner’ом**. Даже когда запрос можно было закрыть только локальным JSON-инструментом, модель иногда все равно выбирала оба инструмента. Например, на запрос сравнения Axum и Actix без явного запроса на live-данные planner все равно пытался вызвать GitHub tool. Это не ломало систему полностью, но приводило к лишним вызовам и делало поведение менее аккуратным. Чтобы это смягчить, я добавил простой post-filter на стороне backend: если в запросе нет слов, указывающих на GitHub, репозитории, активность, текущие данные или latest/current context, GitHub tool отбрасывается. Это не “идеальный интеллект”, а скорее прагматичное ограничение, но для проекта с лимитированым временем он оказался полезен.

Второй заметный failure case был в **synthesizer’е**. Даже когда tools возвращали ограниченный набор фактов, модель иногда начинала делать слишком уверенные выводы, которых не было напрямую в результатах. Например, она могла переусердствовать в формулировках, вводить неявные “оценки” или делать слишком сильные выводы об экосистеме по ограниченному результату поиска на GitHub. Это особенно заметно в задачах сравнения, где LLM естественно тянет к красивому обобщению. Я частично исправил это более жестким system prompt’ом для synthesizer’а, где явно запретил выдумывать репозитории, scores и unsupported claims. Но полностью проблема не ушла: это типичное ограничение генеративной модели, если не строить еще более строгую схему synthesis.

Еще одна реальная трудность была чисто инженерной. Пока проект поднимался в Docker, возникали проблемы с несовместимостью `glibc`, отсутствием `ca-certificates` в runtime-образе, неправильным адресом Ollama внутри Docker-сети и ошибками в инициализации модели. Отдельно всплыл баг, когда endpoint `/history/{session_id}` был привязан не к тому handler’у и вместо истории фактически создавал новую сессию. Это был хороший пример того, что в агентных проектах много проблем не только в LLM-части, но и в обычной инфраструктуре: маршрутизация, конфиг, контейнеризация, сети и правильная передача переменных окружения могут ломать систему не хуже, чем неудачный prompt.

## Что бы я сделал дальше, если бы было больше времени

Если бы у меня было больше времени, я бы в первую очередь усилил **контроль над planner’ом и synthesizer’ом**. Для planner’а я бы сделал более строгую схему выбора инструментов: возможно, не только через prompt, но и через более явные правила маршрутизации, которые заранее определяют тип запроса. Сейчас planner уже работает, но он все еще слегка избыточен. Для synthesizer’а я бы сделал более строгий grounded output: например, заставлял бы его ссылаться только на конкретные фрагменты tool outputs и отдельно разделять stable facts из локальной базы и live facts из GitHub.

Второе направление — улучшение **retrieval по локальному JSON**. Сейчас там используется понятный и объяснимый keyword-based scoring, что хорошо для MVP, но плохо работает на более сложных формулировках и синонимах. Я бы добавил расширение запроса через словарь синонимов или embeddings-based retrieval, чтобы агент лучше понимал, что “observability”, “instrumentation” и “tracing” могут быть семантически связаны.

Третье направление — **оценка качества**. Сейчас проект можно тестировать вручную через набор сценариев, но если делать его серьезнее, я бы добавил evaluation harness: список тестовых запросов и ожидаемых свойств поведения. Например: должен ли вызываться GitHub tool, должен ли ответ быть без инструментов, не упомянула ли модель repo, которого не было в tool output, и так далее. Это помогло бы оценивать изменения не по ощущениям, а более формально.

Также я бы улучшил **frontend**: добавил бы streaming ответов, более удобный просмотр истории, отдельный блок с tool traces и, возможно, переключатель debug mode, чтобы можно было видеть planner output и raw tool results прямо из интерфейса. Сейчас Streamlit frontend достаточно хороший для демо, но он еще не раскрывает весь внутренний pipeline так прозрачно, как мог бы.

И наконец, я бы добавил еще один-два инструмента, например поиск по crates ecosystem или более узкий API для Rust package metadata, чтобы агент мог сравнивать не только репозитории, но и, например, библиотечную зрелость и документацию. Это сделало бы систему богаче и еще лучше показало бы agentic design как подход, а не просто набор отдельных endpoint’ов.
